PROJECT OVERVIEW
================================================================================

Vera is a sophisticated AI-powered data analysis platform that bridges the gap
between data science and practical data interpretation. The platform combines
automated statistical analysis with Large Language Model (LLM) reasoning to
provide enterprise-grade data insights.

CORE PHILOSOPHY: "Python does the math. AI does the thinking."

The system leverages Python's scientific libraries (NumPy, Pandas, Scikit-Learn)
for accurate computations and uses OpenAI-compatible APIs (OXLO) for contextual
reasoning and human-interpreted insights.

3. DATA FLOW ARCHITECTURE
   ──────────────────────

   STEP 1: DATA UPLOAD
   ──────────────────
   User uploads CSV → Flask receives file → DataIngestion validates format →
   → Stores in MongoDB GridFS with metadata → Returns list of all files
   
   Storage: MongoDB GridFS with automatic filename indexing
   Output: Preview (first 5 rows) + Column names

   
   STEP 2: DATASET ANALYSIS
   ───────────────────────
   User clicks "Dataset Info" → Flask loads CSV from GridFS →
   → AnalysisExplainer computes statistics (100% Pandas-based) →
   → LLM receives statistics + null values + unique counts →
   → LLM generates structured insights (JSON) →
   → Insights stored in MongoDB dataset_insights collection
   
   Metrics computed: shape, dtypes, null counts, duplicate rows,
                     numeric stats (mean/std/min/max), unique values,
                     memory usage, data types, column-by-column analysis


   STEP 3: PREPROCESSING & EDA
   ───────────────────────────
   User configures preprocessing → Flask calls DataPreprocessing →
   → Applies: duplicates removal, column dropping, missing value strategies →
   → Generates 25+ EDA charts (histograms, distributions, correlations) →
   → Charts converted to base64 PNG and sent to frontend
   
   On-demand: User clicks "AI Insights" on chart → Frontend sends base64 +
   title → Backend calls OXLO vision API → Returns JSON analysis →
   → Chart insights stored in MongoDB → Frontend displays


   STEP 4: MACHINE LEARNING
   ────────────────────────
   User selects target column → Flask calls MLPipeline →
   → LLM decides feature engineering (drop/ordinal/onehot/numeric) →
   → Auto-detects problem type (classification vs regression) →
   → Trains 10+ model families (LR, DT, RF, GB, SVM, KNN, NB, XGB, etc.) →
   → Performs cross-validation & hyperparameter tuning →
   → Selects best model based on evaluation metric →
   → Saves all models (.pkl) to GridFS + metrics to MongoDB →
   → Returns full results page with confusion matrix, feature importance, etc.

AI/LLM INTEGRATION
================================================================================

1. OXLO API INTEGRATION
   ─────────────────────

   Provider: OXLO (OpenAI-compatible API)
   Models available:
   ├── llama-3.2-3b (For dataset analysis & feature planning)
   ├── mistral-7b (For chart vision analysis)
   └── Support for multimodal inputs (text + vision)

   Configuration:
   ├── Environment variable: OXLO_API_KEY
   ├── Base URL: https://api.oxlo.ai/v1
   ├── Auth: Bearer token in API key
   └── Timeout: 90 seconds per request

   Integration points:
   ├── /info route - LLM analyzes dataset statistics
   ├── /preprocessing route - LLM suggests missing value strategies
   ├── /ml route - LLM decides feature engineering approach
   └── /analyse_chart route - LLM analyzes chart images


2. PROMPT ENGINEERING
   ───────────────────

   Dataset Analysis Prompt:
   - INPUT: DataFrame statistics (shape, dtypes, nulls, unique counts)
   - OUTPUT: {summary, quality_flags, column_insights, next_steps}
   - The prompt is structured to ensure JSON-only responses (no preamble)
   - Includes uncertainty notation when confidence is low

   Feature Engineering Prompt:
   - INPUT: Column names + cardinality + target variable
   - OUTPUT: {drop: [], ordinal: {}, onehot: [], numeric: []}
   - Rules: never include target, drop high-cardinality, etc.
   - Helps ML pipeline make intelligent preprocessing decisions

   Chart Analysis Prompt:
   - INPUT: Base64 PNG image + chart title
   - OUTPUT: {represents, key_findings[], anomalies[], recommendations[]}
   - Uses vision capabilities to interpret chart patterns
   - Provides actionable recommendations based on chart data


3. ERROR HANDLING & FALLBACKS
   ─────────────────────────────

   API Failures:
   ├── 429 (Rate limit): Exponential backoff (1s, 2s, 4s, 8s max)
   ├── 401 (Auth error): Returns user-facing error message
   ├── Timeout: Returns empty analysis placeholder
   └── Network error: Gracefully degrades to empty response

   JSON Parsing:
   ├── Invalid JSON: Attempts markdown fence removal
   ├── Partial response: Uses regex to extract valid JSON substring
   ├── Complete failure: Returns empty placeholder (no error propagation)
   └── Logged for debugging but doesn't crash system

   User Experience:
   ├── Failed analysis shows: "Analysis unavailable - ensure OXLO_API_KEY is configured"
   ├── Includes retry button for on-demand re-analysis
   ├── Platform continues functioning without LLM


DATABASE ARCHITECTURE
================================================================================

1. MONGODB SCHEMA
   ───────────────

   Collection: fs.files (GridFS metadata)
   ├── _id: ObjectId (auto)
   ├── filename: String (e.g., "Churn_Modelling.csv")
   ├── uploadDate: DateTime (automatic)
   ├── content_type: String (e.g., "text/csv")
   └── length: Integer (file size in bytes)

   Collection: fs.chunks (GridFS data)
   ├── _id: ObjectId (auto)
   ├── files_id: ObjectId (reference to fs.files)
   ├── n: Integer (chunk sequence number)
   └── data: BinData (binary content)

   Collection: dataset_insights
   ├── _id: ObjectId (auto)
   ├── filename: String (unique, identifies dataset)
   ├── stored_at: DateTime (when analysis was performed)
   ├── analysis: Object
   │   ├── shape: [rows, columns]
   │   ├── columns: [column names]
   │   ├── dtypes: {column: dtype}
   │   ├── null_values: {column: count}
   │   ├── null_percentages: {column: percentage}
   │   ├── duplicate_rows: Integer
   │   ├── numeric_columns: [names]
   │   ├── categorical_columns: [names]
   │   ├── describe: {column: {mean, std, min, 25%, 50%, 75%, max}}
   │   ├── memory_usage_mb: Float
   │   ├── unique_counts: {column: count}
   │   └── sample_rows: [preview data]
   ├── ai_insights: Object
   │   ├── summary: String (overall dataset assessment)
   │   ├── quality_flags: [issues found]
   │   ├── column_insights: {column: analysis}
   │   └── next_steps: [recommended actions]
   ├── unique: {column: [unique values sample]}
   └── chart_insights: {chart_title: {represents, key_findings, anomalies, recommendations}}

   Collection: ml_results
   ├── _id: ObjectId (auto)
   ├── filename: String
   ├── target_column: String
   ├── created_at: DateTime
   ├── problem_type: String ("classification" or "regression")
   ├── feature_plan: Object (LLM's feature engineering decisions)
   ├── best_model: String (model name)
   ├── best_metric: String (metric name)
   ├── rank_metric: Float (best metric value)
   ├── results: Object
   │   ├── LogisticRegression: {accuracy, f1, precision, recall, auc}
   │   ├── RandomForest: {...}
   │   ├── XGBoost: {...}
   │   └── ... (for all trained models)
   ├── confusion_matrix: Array (for classification)
   ├── feature_importance: {feature: importance_score}
   ├── model_gridfs_ids: {model_name: gridfs_id}
   └── training_time_seconds: Float


2. DATA PERSISTENCE STRATEGY
   ──────────────────────────

   CSV File Storage:
   ├── Location: MongoDB GridFS
   ├── Indexing: By filename (unique)
   ├── Retrieval: Latest by uploadDate
   └── Retrieval: By specific filename

   Model Storage:
   ├── Format: Pickle (.pkl) binary
   ├── Location: MongoDB GridFS
   ├── Referenced by: ml_results.model_gridfs_ids
   └── Download: Via /download_model endpoint

   Analysis Caching:
   ├── Dataset insights cached in dataset_insights collection
   ├── Updates: Latest analysis overwrites previous
   ├── TTL: No automatic expiration (manual cleanup possible)
   └── Retrieval: Via /api/insights/<filename>

DATA FLOW EXAMPLES
================================================================================

1. COMPLETE WORKFLOW: CSV UPLOAD TO ML RESULTS
   ──────────────────────────────────────────

   1.1 FILE UPLOAD
       └─ POST /upload with CSV file
          ├─ Flask receives file
          ├─ DataIngestion validates format (charset, separator)
          ├─ GridFS stores with filename index
          └─ Redirect to /upload?filename=XXX

   1.2 DATASET PREVIEW
       └─ GET /upload?filename=Churn_Modelling.csv
          ├─ Flask loads file list
          ├─ Shows selected file preview (first 5 rows)
          └─ User clicks "Dataset Info"

   1.3 ANALYSIS GENERATION
       └─ GET /info?filename=Churn_Modelling.csv
          ├─ AnalysisExplainer loads DataFrame from GridFS
          ├─ compute_analysis() calculates statistics (100% Pandas)
          ├─ Sends structured JSON to OXLO API with statistics
          ├─ Receives back: {summary, quality_flags, column_insights}
          ├─ store_dataset_insights() saves to MongoDB
          └─ Template renders with all insights

   1.4 PREPROCESSING CONFIGURATION
       └─ GET /preprocessing?filename=Churn_Modelling.csv
          ├─ DataPreprocessing loads DataFrame
          ├─ get_ai_insights() calls LLM for missing value strategies
          ├─ Template shows form with LLM recommendations
          └─ User submits form with selections

   1.5 EDA EXECUTION & VISUALIZATION
       └─ POST /preprocessing (form data)
          ├─ DataPreprocessing applies transformations:
          │  ├─ Remove duplicates
          │  ├─ Drop selected columns
          │  └─ Handle missing values per strategy
          ├─ generate_eda_report() creates 25 charts:
          │  ├─ Each chart rendered with Matplotlib
          │  ├─ Each chart converted to base64 PNG
          │  └─ All sent to template as {title: base64}
          ├─ Template displays charts in responsive grid
          └─ User clicks "AI Insights" on any chart

   1.6 ON-DEMAND CHART ANALYSIS
       └─ POST /analyse_chart
          ├─ JavaScript sends: {image_b64, chart_title, filename}
          ├─ analyse_chart() calls OXLO vision API
          ├─ Returns: {represents, key_findings[], anomalies[], recommendations[]}
          ├─ store_chart_insight() saves to MongoDB
          ├─ JavaScript displays analysis panel
          └─ User can retry different charts

   1.7 ML PIPELINE CONFIGURATION
       └─ GET /ml?filename=Churn_Modelling.csv
          ├─ Flask loads column list from cleaned DataFrame
          ├─ Template shows target column dropdown
          └─ User selects target and submits

   1.8 ML PIPELINE EXECUTION
       └─ POST /ml (form: target_column)
          ├─ MLPipeline.llm_feature_plan() asks LLM:
          │  └─ What columns to drop/ordinal/onehot/numeric
          ├─ auto_detect_problem() determines classification vs regression
          ├─ Applies preprocessing transformer:
          │  ├─ StandardScaler for numeric features
          │  ├─ OrdinalEncoder for ordinal features
          │  └─ OneHotEncoder for categorical features
          ├─ train_all_models() trains 10+ model families:
          │  ├─ Each with cross-validation (5-fold)
          │  ├─ Each with hyperparameter tuning
          │  └─ Each evaluated on validation metrics
          ├── select_best_model() picks winner based on metric
          ├── save_results() to MongoDB:
          │   ├─ All model metrics
          │   ├─ Confusion matrix / residuals
          │   ├─ Feature importance
          │   └─ Pickled models to GridFS
          └─ Template renders results page

   1.9 MODEL DOWNLOAD
       └─ GET /download_model?doc_id=XXX&model_name=RandomForest
          ├─ Retrieves model ObjectID from MongoDB
          ├─ Gets pickle bytes from GridFS
          ├─ Sends as file download
          └─ User has trained model for local use

DEBUGGING & TROUBLESHOOTING
================================================================================

1. COMMON ISSUES
   ──────────────

   Issue: "OXLO_API_KEY not configured"
   ─────────────────────────────────────
   Cause: .env file missing or OXLO_API_KEY not set
   Solution:
     1. Create .env in project root
     2. Add: OXLO_API_KEY=your_actual_key
     3. Restart Flask
     4. Browser console should show no errors

   Issue: MongoDB connection errors
   ────────────────────────────────
   Cause: MongoDB not running on localhost:27017
   Solution:
     1. Verify MongoDB service running
     2. Check connection string in code
     3. Verify firewall allows port 27017
     4. Test: mongosh localhost:27017

   Issue: Chart analysis shows "Retry" button
   ──────────────────────────────────────────
   Cause: OXLO API key invalid, rate limited, or network error
   Solution:
     1. Check browser console (F12 → Console)
     2. Check Flask logs (flask terminal output)
     3. Verify OXLO_API_KEY is valid
     4. Check internet connectivity
     5. Retry after rate limit window

   Issue: Files not uploading
   ─────────────────────────
   Cause: File too large, unsupported format, encoding issue
   Solution:
     1. Verify CSV format (comma-separated)
     2. Check file is < 50MB
     3. Ensure UTF-8 encoding
     4. Try with sample data first

   Issue: ML pipeline takes very long
   ──────────────────────────────────
   Cause: Large dataset, many features, long hyperparameter search
   Solution:
     1. Reduce dataset size (remove rows)
     2. Drop unnecessary columns
     3. Reduce cross-validation folds
     4. Skip hyperparameter tuning


2. DEBUG MODE CHECKLIST
   ─────────────────────

   Step 1: Verify Environment
   ├─ Check .env file exists
   ├─ Verify OXLO_API_KEY is set
   ├─ Check MongoDB running
   └─ View Flask logs in terminal

   Step 2: Check Browser Console
   ├─ F12 → Console tab
   ├─ Look for JavaScript errors
   ├─ Check Network tab for API failures
   └─ Verify API responses in Response tab

   Step 3: Check Flask Logs
   ├─ Look for exception messages
   ├─ Check MongoDB connection logs
   ├─ Verify OXLO API calls succeeded (200 OK)
   └─ Check error response content

   Step 4: Verify MongoDB Data
   ├─ mongosh localhost:27017
   ├─ use clarityAI_database
   ├─ db.fs.files.find() - check uploaded files
   ├─ db.dataset_insights.find() - check analysis results
   └─ db.ml_results.find() - check ML results

   Step 5: Test API Endpoints Manually
   ├─ curl http://localhost:5001/api/health
   ├─ curl http://localhost:5001/upload (GET)
   ├─ Verify responses in terminal
